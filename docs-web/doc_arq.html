Proposta Arquitetônica

Visão completa da arquitetura do sistema IoT Gateway

Visão Geral

O sistema IoT Gateway é uma solução modular para integração de múltiplos protocolos de sensores IoT, centralizando a comunicação através do protocolo MQTT e fornecendo uma interface unificada para processamento de dados.

Camadas da Arquitetura (5 Camadas)

1. Camada de Sensores (Física)

  • 6 Sensores Ativos: LoRa (2), Zigbee (1), Sigfox (1), RFID (2)
  • LoRa: Longo alcance, baixo consumo (Entrada Principal, Biblioteca)
  • Zigbee: Redes mesh, automação (Quadra)
  • Sigfox: Comunicação global (Refeitório)
  • RFID: Identificação (Sala 101, Sala 102)

2. Camada de Gateway (IoT)

  • MQTT Client (Paho): Publicação em sensores/+/dados
  • Message Formatter: JSON padronizado com timestamp
  • Config Loader: Leitura de mqtt_config.ini
  • Gateway Core: Orquestração dos 6 sensores ativos
  • 6.026 mensagens publicadas desde início do sistema

3. Camada de Broker MQTT

  • Eclipse Mosquitto 2.0.18: Broker local (localhost:1883)
  • Topics: sensores/{protocol}/dados (lora, zigbee, sigfox, rfid)
  • QoS Level 1: At least once delivery
  • Subscriber: Gateway consome e persiste no MySQL

4. Camada de Backend (API REST)

  • Flask 3.0: 29 endpoints REST ativos (localhost:5000)
  • SQLAlchemy ORM: Mapeamento objeto-relacional
  • Marshmallow: Validação e serialização (4 schemas)
  • JWT Extended: Autenticação com tokens 24h
  • Services Layer: Lógica de negócio isolada
  • Utils Layer: Error handlers, decorators, validators

5. Camada de Dados (Persistência)

  • MySQL 8.4.6: Banco relacional (localhost:3306)
  • Database: ceu_tres_pontes_db
  • 5 Tabelas: sensors, readings, users, alerts, statistics
  • 6.026 leituras armazenadas com timestamps
  • Índices otimizados: sensor_id, timestamp, activity
  • JSON Fields: Metadados flexíveis (protocol_config, sensor_metadata)

Início / Integrações

Integrações

Protocolos suportados e suas características

LoRa (Long Range)

Características:

  • Alcance: até 15 km (rural)
  • Frequência: 868 MHz (EU) / 915 MHz (US)
  • Taxa: 0.3 - 50 kbps
  • Consumo: Muito baixo

Uso: Sensores remotos, agricultura, smart cities

Topic MQTT: sensores/lora/dados

Zigbee

Características:

  • Alcance: 10-100 metros
  • Frequência: 2.4 GHz
  • Taxa: 250 kbps
  • Topologia: Mesh

Uso: Automação residencial, iluminação inteligente

Topic MQTT: sensores/zigbee/dados

Sigfox

Características:

  • Alcance: até 50 km
  • Frequência: 868 MHz
  • Taxa: 100 bps
  • Mensagens: 140 msg/dia

Uso: Rastreamento, medição, alarmes

Topic MQTT: sensores/sigfox/dados

RFID

Características:

  • Alcance: 0.1 - 10 metros
  • Frequência: 125 kHz - 2.4 GHz
  • Taxa: Instantânea
  • Sem bateria (passivo)

Uso: Controle de acesso, inventário, logística

Topic MQTT: sensores/rfid/dados

Tabela: sensors

CREATE TABLE sensors (
  id INT PRIMARY KEY AUTO_INCREMENT,
  serial_number VARCHAR(50) UNIQUE NOT NULL,
  protocol ENUM('LoRa', 'ZigBee', 'Sigfox', 'RFID') NOT NULL,
  location VARCHAR(200) NOT NULL,
  description TEXT,
  status ENUM('active', 'inactive', 'maintenance', 'error') DEFAULT 'active',
  protocol_config JSON,
  firmware_version VARCHAR(20),
  battery_level INT,
  signal_strength INT,
  total_readings INT DEFAULT 0,
  last_reading_at DATETIME,
  created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
  updated_at DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP
);

-- Exemplo de registro:
{
  "id": 1,
  "serial_number": "LORA-ENTRADA-01",
  "protocol": "LoRa",
  "location": "Entrada Principal",
  "status": "active",
  "battery_level": 87,
  "signal_strength": -75,
  "total_readings": 826
}

Tabela: readings

CREATE TABLE readings (
  id BIGINT PRIMARY KEY AUTO_INCREMENT,
  sensor_id INT NOT NULL,
  activity SMALLINT NOT NULL,  -- 0 = nenhuma, 1 = detecção
  timestamp DATETIME NOT NULL,
  sensor_metadata JSON,        -- battery_level, signal_strength, etc
  message_id VARCHAR(100),
  gateway_id VARCHAR(50),
  created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
  FOREIGN KEY (sensor_id) REFERENCES sensors(id) ON DELETE CASCADE,
  INDEX idx_sensor_timestamp (sensor_id, timestamp),
  INDEX idx_timestamp_activity (timestamp, activity)
);

-- Exemplo de registro:
{
  "id": 1,
  "sensor_id": 1,
  "activity": 1,
  "timestamp": "2025-10-15T14:30:00",
  "sensor_metadata": {
    "battery_level": 85,
    "signal_strength": -72,
    "temperature": 23.5
  }
}

Tabela: users

CREATE TABLE users (
  id INT PRIMARY KEY AUTO_INCREMENT,
  username VARCHAR(50) UNIQUE NOT NULL,
  email VARCHAR(100) UNIQUE NOT NULL,
  password_hash VARCHAR(255) NOT NULL,
  full_name VARCHAR(100),
  phone VARCHAR(20),
  role ENUM('admin', 'operator', 'viewer') DEFAULT 'operator',
  is_active BOOLEAN DEFAULT TRUE,
  is_verified BOOLEAN DEFAULT FALSE,
  last_login DATETIME,
  login_count INT DEFAULT 0,
  created_at DATETIME DEFAULT CURRENT_TIMESTAMP,

-- Usuário padrão:
{
  "username": "admin",
  "email": "admin@ceutrespontes.com",
  "role": "admin",
  "login_count": 14
}

Relacionamentos

  • sensors → readings: Um sensor tem muitas leituras (1:N)
  • sensors → alerts: Um sensor pode ter muitos alertas (1:N)
  • Cascade Delete: Ao deletar sensor, deleta leituras e alertas
  • JSON Fields: protocol_config e sensor_metadata para dados flexíveis

Segurança

  • Session Management: Controle de login_count e last_login

  • Marshmallow Schemas: Validação estrita de entrada (4 schemas)
  • Input Sanitization: @validate_request decorator personalizado
  • SQL Injection Prevention: SQLAlchemy ORM com prepared statements
  • JSON Validation: Formato de mensagens MQTT e API
  • Email Validation: Formato RFC-compliant

Segurança de Infraestrutura

  • MQTT Local Network: Comunicação broker em rede local
  • Database Security: Conexões autenticadas, usuário dedicado
  • Error Handling: Mensagens genéricas, logs detalhados
  • CORS Configuration: Controle de origens permitidas

Monitoramento da Piscina

Visão Geral

Sistema completo de monitoramento da piscina do CEU Três Pontes, com coleta de dados de temperatura e qualidade da água em tempo real, armazenamento em banco de dados, e visualização através de dashboard interativo.

1. Objetivos

  • Monitorar continuamente a temperatura da água da piscina
  • Monitorar a temperatura ambiente ao redor da piscina
  • Avaliar e registrar a qualidade da água (Ótima, Boa, Regular, Imprópria)
  • Gerar alertas quando a qualidade da água não estiver adequada
  • Fornecer histórico de médias diárias de temperatura
  • Apresentar estatísticas agregadas do período

2. Atores

  • Administrador do Sistema: Gerencia usuários e configurações
  • Operador do CEU: Monitora os dados da piscina e responde a alertas
  • Sensores IoT: Coletam dados de temperatura e qualidade da água
  • Sistema de Alertas: Notifica sobre condições inadequadas da água

3.2 Fluxo de Dados

  2. Transmissão: Dados enviados via MQTT para tópicos específicos
  3. Processamento: Gateway recebe, formata e envia para API
  4. Armazenamento: API valida e persiste dados no MySQL
  5. Disponibilização: API expõe endpoints REST autenticados (JWT)
  6. Visualização: Dashboard consulta API e renderiza gráficos
  7. Alertas: Sistema verifica qualidade e gera alertas quando necessário

4. Modelo de Dados

4.1 Tabela: pool_readings

CREATE TABLE pool_readings (
    id INT AUTO_INCREMENT PRIMARY KEY,
    sensor_type VARCHAR(20) NOT NULL,           -- 'water_temp', 'ambient_temp', 'water_quality'
    reading_date DATE NOT NULL,                 -- Data da leitura
    reading_time TIME NOT NULL,                 -- Hora da leitura (timezone local)
    temperature DECIMAL(5,2),                   -- Temperatura em °C (2 decimais)
    water_quality VARCHAR(20),                  -- 'Ótima', 'Boa', 'Regular', 'Imprópria'
    created_at DATETIME NOT NULL,               -- Timestamp de criação do registro
    updated_at DATETIME NOT NULL,               -- Timestamp de atualização
    INDEX idx_sensor_type (sensor_type),
    INDEX idx_reading_date (reading_date)
);

4.2 Tipos de Sensores

  • water_temp: Temperatura da água da piscina (20-35°C)
  • ambient_temp: Temperatura ambiente (15-40°C)
  • water_quality: Qualidade da água (sem temperatura)

  • Cards de Status: Exibem última leitura de cada sensor em tempo real
  • Gráficos de Histórico: Média diária dos últimos 10 dias (Chart.js)
  • Estatísticas do Período: Médias, mínimas e máximas
  • Lista de Alertas: Histórico de alertas de qualidade inadequada
  • Atualização Automática: Dados atualizados a cada 30 segundos

6.2 Componentes Visuais

  • Card Temperatura da Água: Valor atual + timestamp formatado (DD/MM/YYYY HH:MM)
  • Card Temperatura Ambiente: Valor atual + timestamp
  • Card Qualidade da Água: Status com código de cores (verde/amarelo/laranja/vermelho)
  • Gráfico Temperatura da Água: Linha azul com médias diárias
  • Gráfico Temperatura Ambiente: Linha laranja com médias diárias
  • Tabela de Alertas: Lista filtrada das últimas 24h

6.3 Formatação de Dados

  • Temperaturas: 2 casas decimais (ex: 25.46°C)
  • Datas: Formato brasileiro DD/MM/YYYY
  • Horários: HH:MM (24h)
  • Timezone: America/Sao_Paulo (UTC-3)

7. Decisões Técnicas

7.1 Timezone

Problema: Inicialmente, as leituras eram geradas em UTC, causando diferença de 3 horas com o horário local do servidor.

Solução: Implementação do uso de zoneinfo.ZoneInfo('America/Sao_Paulo') em todos os scripts de geração de dados.

Resultado: Dados armazenados com timestamp local correto, sem necessidade de conversão no frontend.

7.2 Média Diária vs. Leituras Horárias

Problema: Gráficos com leituras individuais (hora a hora) ficavam poluídos e difíceis de interpretar.

Solução: Criação de endpoint dedicado (/temperature/daily-average) que calcula média de todas as leituras de cada dia.

Resultado: Gráficos limpos e informativos mostrando tendências ao longo de 10 dias.

7.3 Separação de Date e Time

Decisão: Usar campos separados reading_date (DATE) e reading_time (TIME) ao invés de um único DATETIME.

Justificativa: Facilita queries de agregação por dia e evita complexidades de timezone em campos DATETIME do MySQL.

Trade-off: Necessário combinar os dois campos para exibição, mas queries de média diária ficam mais simples.

8. Melhorias Implementadas

8.1 Correção de Horários Futuros

Script de população ajustado para não gerar leituras com timestamp futuro no dia atual:

  • Dias passados: Leituras entre 6h e 22h (horário de funcionamento da piscina)
  • Dia atual: Leituras apenas até a hora atual do servidor
  • Quantidade proporcional: Número de leituras de hoje ajustado conforme horas decorridas

8.2 Formatação de Temperatura

Padronização de todas as temperaturas para exatamente 2 casas decimais:

  • Banco de dados: DECIMAL(5,2)
  • API: Conversão para float com arredondamento
  • Frontend: parseFloat(value).toFixed(2)

8.3 Cache Control

Configuração do NGINX para evitar cache de páginas HTML:

add_header Cache-Control "no-store, no-cache, must-revalidate, max-age=0";
add_header Pragma "no-cache";
